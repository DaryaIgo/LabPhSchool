# Памятка по развёртыванию Ph на SprintHost

## Как мы работаем с деплоем

1. **Архив `deploy-minimal.tar.gz` содержит только код приложения**:
   - собранный фронтенд (`dist/public/`);
   - собранный бэкенд (`dist/boot.cjs`);
   - `.env.production`;
   - `.htaccess`;
   - `tmp/restart.txt` — триггер для перезапуска Passenger;
   - вспомогательные SQL-дампы (отдельно, см. ниже).

   Архив **не содержит** `node_modules` (они не нужны, бэкенд собран в один файл) и **не содержит** дампов базы данных (их удобнее импортировать отдельно через PHPMyAdmin).

2. **Загрузка архива**:
   - загрузить `deploy-minimal.tar.gz` в `public_html`;
   - распаковать содержимое в `/home/darigoshin/domains/nebulus.ru/app/`;
   - скопировать `.htaccess` из `app/` в `public_html/`;
   - дождаться перезапуска Passenger (файл `tmp/restart.txt` в архиве должен инициировать перезапуск).

3. **База данных импортируется отдельно через PHPMyAdmin**, а не из архива. Это удобнее, потому что:
   - SQL-файлы можно импортировать выборочно;
   - легче избежать ошибок дублирования ключей;
   - можно проверить результат до того, как Passenger подхватит новую версию приложения.

## Архитектура базы данных

Проект перешёл на модульную (domain-driven) организацию данных. Все таблицы логически разбиты на домены:

| Домен | Таблицы |
|---|---|
| `auth` | `roles`, `permissions`, `role_permissions`, `users`, `local_users` |
| `content` | `topic_nodes`, `resources` |
| `learning` | `enrollments`, `student_progress`, `lab_progress` |
| `labs` | `lab_categories`, `lab_subcategories`, `lab_works`, `lab_blocks`, `lab_simulation_params`, `lab_analytics` |
| `problems` | `problem_types`, `problems` |
| `jupyter` | `jupyter_notebooks`, `jupyter_notebook_access` |
| `notifications` | `notifications` |
| `timeline` | `timeline_entries` |
| `audit` | `audit_log` |
| `media` | `images` |

> **Важно:** таблицы `topics`, `subtopics` и устаревшая таблица `labs` были удалены. `topic_nodes` — единственный источник правды для структуры курса. Корневые узлы — темы курса, дочерние узлы — подтемы/уроки.

На текущем хостинге SprintHost физически используется одна база данных (`darigoshin_labphschool_bd`). В коде каждый домен подключается через собственную фабрику `get*Db()`, но по умолчанию они все используют переменную `DATABASE_URL`. При необходимости каждому домену можно задать отдельную базу через `DATABASE_URL_<DOMAIN>` (см. `app/.env.example`).

Между доменами не осталось жёстких внешних ключей: связи реализованы через soft-ссылки (`id` + домен). Это позволяет в будущем разнести домены по разным физическим базам без изменения кода приложения.

## Миграции

**На хостинге миграции не применяются.** База данных пересоздаётся заново через PHPMyAdmin (удаление старых таблиц и импорт актуальных SQL-дампов). При локальной разработке можно использовать `npm run db:push` или накатывать миграции с нуля.

## Файлы с данными для БД

| Файл | Что содержит | Когда импортировать |
|------|--------------|---------------------|
| `db/admin-seed.sql` | роли `admin`/`student`/`teacher` и пользователь `admin` | один раз при первой настройке |
| `db/virtual-lab-data.sql` | категории, подкатегории, лабораторные работы, блоки, параметры симуляций | один раз, если таблицы `lab_*` пустые |
| `db/timeline-data.sql` | учёные и открытия для таймлайна (`timeline_entries`) | один раз, если `timeline_entries` пустая |
| `db/problems-data.sql` | типы задач (`problem_types`) и сами задачи (`problems`) | один раз, если таблицы `problem_types`/`problems` пустые |
| `db/topic-nodes-data.sql` | полный контент тем и подтем (`topic_nodes`) | если курс отображается в сокращённом виде / после пересоздания таблиц |
| `db/image-storage-migration.sql` | таблица `images` для хранения загруженных картинок | при обновлении на версию, где картинки хранятся в БД |

> `db/subtopics-jupyter-update.sql` больше не используется — ссылки Jupyter теперь хранятся в колонке `topic_nodes.ipynb_url`.

## Как импортировать SQL через PHPMyAdmin

1. Открыть панель SprintHost → «Базы данных» → «Войти в PHPMyAdmin».
2. Выбрать базу `darigoshin_labphschool_bd`.
3. Перейти на вкладку **«Импорт»**.
4. Выбрать нужный SQL-файл.
5. Нажать **«Вперёд»**.

### Ошибка дублирования ключа

Если появляется:

```
#1062 - Дублирующаяся запись '1' по ключу 'lab_categories.PRIMARY'
```

значит, эти данные уже импортированы. Не нужно импортировать их повторно. Импортируйте только те файлы, таблицы которых ещё пусты (например, `problems-data.sql`, если `timeline_entries` уже заполнена).

## Проверка после импорта

- **Админ-панель**: таблицы `roles` и `local_users` должны содержать запись `admin`.
- **Лаборатории**: таблицы `lab_categories`, `lab_works` и т.д. должны быть заполнены.
- **Таймлайн**: таблица `timeline_entries` должна быть заполнена.
- **Курсы**: таблица `topic_nodes` должна содержать `content` с Markdown.
- **Задачи**: таблицы `problem_types`, `problems` должны быть заполнены.

## Если что-то не отображается на сайте

1. **Очистить куки и кэш браузера** для тестового домена.
2. **Подождать 10–20 секунд** — Passenger может ещё не перезапустить приложение после загрузки файлов.
3. **Открыть F12 → Console** и посмотреть ошибки JavaScript.
4. **Открыть F12 → Network** и проверить, не падают ли запросы к `/api/trpc/`.
5. Если запрос падает с ошибкой 500, проверить логи Passenger в панели управления SprintHost.

## План деплоя при обновлении

При обновлении кода на хостинге **без выполнения команд на сервере**:

1. **Импортировать SQL-миграцию через PHPMyAdmin**:
   - файл `db/image-storage-migration.sql`;
   - создаёт таблицу `images`, в которой будут храниться загруженные картинки.
2. **Загрузить и распаковать свежий архив `deploy-minimal.tar.gz`** в `/home/darigoshin/domains/igoshina-line.ru/app/`.
3. **Скопировать `.htaccess`** из `app/` в `public_html/`.
4. **Дождаться**, пока Passenger автоматически перезапустит приложение.

После этого картинки можно добавлять через админку прямо на сервере — они будут сохраняться в базу данных и не пропадут при следующем деплое.

## Ограничения хостинга и правила деплоя

На SprintHost **нельзя выполнять команды на сервере** (нет SSH-доступа для запуска скриптов). Поэтому:

- Все изменения в БД вносятся только через **PHPMyAdmin** → Импорт SQL.
- Архив `deploy-minimal.tar.gz` загружается и распаковывается через файловый менеджер хостинга.
- Не создавать служебных скриптов (`*.sh`, `*.mjs`, `*.ts`), которые предполагают запуск на сервере через SSH или консоль хостинга. Если задачу нельзя решить через SQL-импорт + замену файлов — нужно искать другое решение или согласовывать его с владельцем проекта.

## Известные особенности

- Изображения учёных на таймлайне загружаются с Википедии (внешние URL). Если картинка не грузится, возможна блокировка хотлинка со стороны Wikimedia; в таком случае изображение просто скрывается.
- Контент курсов рендерится Markdown-парсером с поддержкой формул KaTeX. Если текст есть в БД, но не рендерится, причина обычно в ошибке KaTeX на какой-то формуле.
- Изображения, загруженные через админку, хранятся в таблице `images` и не теряются при redeploy.
